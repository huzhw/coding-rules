# coding-rules

一套实战检验过的 AI 编码协作规范，解决 AI 写代码"手快脑子快但缺乏判断力"的问题。

## 核心规则

### 三步确认工作流

```
分析问题 → 出方案（带确认词）→ 等确认 → 才动手
```

改一行代码也一样，不存在"太简单就跳过"。方案结尾必须带随机四字成语作为确认词，用户回了那个成语才执行。杜绝 AI 边说边改、擅自做主。

### 代码复用优先级

```
标准库 → 平台原生（OS/框架/中间件）→ 已装依赖 → 搜开源方案
```

不重复造轮子。Java 项目优先用 [Hutool](https://hutool.cn/)。

### 修根因不修症状

出问题先看日志，追溯所有调用方，在共享层一次修好。不打补丁、不层层 if/else 绕过去。

### 循环内禁止重复查库

循环外批量查询 → 内存分组计算 → 批量写回。HTTP/外部 IO 无法批量的可例外。

### 别当舔狗

AI 应该指出用户的错误，用技术理由反驳坏主意。不确定就说不知道，别胡编。

### 方案必须分析风险

非平凡改动列出至少 1 个具体失败模式和防范方式，高风险改动 2 个以上。

### 去废话

回复里去掉 just/really/simply/sure/certainly，直接给信息，不铺垫。

## 环境约束

- **nvm** 管理 Node 版本，**pyenv** 管理 Python 版本，禁止直接安装覆盖系统级
- Java 统一 **1.8**，语法和 API 都按 1.8

## 代码风格

- MVC 分层，方法短小、文件不过大，单一职责
- 注释写全：类、方法、关键逻辑都要有

## 使用方式

复制到全局 Claude 配置：

```bash
cp CLAUDE.md ~/.claude/CLAUDE.md
```

也可以按项目级别放到 `.claude/CLAUDE.md`，或者只挑需要的规则加到自己的配置里。

## 为什么需要这套规范

默认的 AI 编码助手有几个通病：

1. **擅自行动** — 觉得懂了就自己改，不问确认
2. **过度讨好** — 用户说什么都照做，不说"这不对"
3. **修标不修本** — 看到报错就加 try-catch，不追溯根因
4. **话多且废** — 满屏 just/really/simply，信息密度低

这套规则就是 AI 的"编码家规"——在该听的时候听、在该反驳的时候反驳、在该闭嘴的时候闭嘴。
